## 一、OLED 显示屏调试复盘

### 1.1 问题现象

屏幕显示乱码，字符无法正常辨认。

### 1.2 调试过程

**第一轮排查（失败）**：误认为问题出在字库数据本身，修改了 `ssd1306.c` 中的 `Font6x8` 和 `Font8x16` 数组。

**第二轮排查（失败）**：继续怀疑字库，从头重写整个驱动文件。

**第三轮排查（找到根本原因）**：发现实际编译的工程在 `e:\aiChat\chaoshengbo\UltraasonicRanger\`，而之前一直在修改 `e:\cubmxproject\UltraasonicRanger\` —— 改错了目录。

### 1.3 三个致命错误

打开实际编译工程中的 `ssd1306.c`，发现了与参考项目（野火 `bsp_i2c_oled.c`）的三大差异：

#### 错误 1：字库数据来源不同

| 对比项 | 实际编译工程（错误） | 参考项目（正确） |
|--------|---------------------|-----------------|
| 来源 | Adafruit 风格字库 | 野火/embedfire 字库 |
| 取模方式 | 不同 | 共阴-列行式-逆向输出 |
| 95个ASCII字符字节 | 全部不同 | 经过实际项目验证 |

**关键认知**：OLED 屏幕上的每一个像素点都是由字节中的位来控制的。不同取模方式会导致同一个字符在屏幕上显示完全不同。必须使用与 OLED 驱动芯片兼容的字库取模方式。

#### 错误 2：帧缓冲内存布局 Bug（最致命）

```c
// 错误代码：
static uint8_t Gram[128][8];  // 列优先存储

void OLED_UpdateScreen(void) {
    for (uint8_t page = 0; page < 8; page++) {
        OLED_SetPos(page, 0);
        // BUG：无论 page 是多少，永远发 Gram[0] 开始的 128 字节！
        HAL_I2C_Mem_Write(&hi2c1, addr, 0x40, I2C_MEMADD_SIZE_8BIT,
                          Gram[0] + page * 128, 128, 100);
    }
}
```

**错误分析**：
- `Gram[0] + page * 128` 在 C 语言中的语义是：`Gram[0]` 的地址（即第一列首地址）加上 `page*128` 字节偏移
- 但 `Gram` 定义是 `uint8_t Gram[128][8]`，列优先存储，一列是 8 字节
- 第 page 页的正确位置是列首地址 + page 字节，而不是 `page*128` 字节
- 导致 OLED 收到的数据全部来自错误的内存地址，显示完全混乱的像素

#### 错误 3：8x16 数字使用拉伸算法而非专用字库

```c
// 错误做法：把 6x8 字库的字符垂直拉伸 2 倍来显示 16px 高
// 这导致数字下半部分完全变形

// 正确做法：使用独立的 8x16 字库（上半页 8 字节 + 下半页 8 字节）
```

### 1.4 正确方案

完全参照参考项目的 `bsp_i2c_oled.c` 重写驱动：

| 改动点 | 错误方式 | 正确方式 |
|--------|---------|---------|
| 架构 | 帧缓冲 + UpdateScreen 刷新 | **直接写 GDDRAM**（实时 I2C 发送） |
| I2C 写入 | `HAL_I2C_Mem_Write` | **`HAL_I2C_Master_Transmit`** |
| 字库 | Adafruit 风格 | **野火验证过的字库**（逐字节复制） |
| 8x16 数字 | 拉伸算法 | **独立 Font8x16[10][16]** |

### 1.5 重点关注

- **改代码前先确认编译的是哪个目录的工程**（这是本次最大教训）
- OLED 驱动开发的核心是字库取模方式必须与显示逻辑一致
- 帧缓冲方案虽灵活但容易出内存布局错误，初学者建议用直接写 GDDRAM 方案
- I2C 地址注意：SSD1306 7 位地址 `0x3C`，HAL 层左移 1 位后是 `0x78`

## 二、SYN6288 语音模块调试复盘

### 2.1 问题现象

喇叭完全不发声，串口调试显示数据已发出。

### 2.2 调试过程

**第一轮排查**：怀疑波特率不对（改成了 115200，参考项目是 9600）。

**第二轮排查**：添加 USART2 硬件回环自检（短接 PA2/PA3），确认 USART2 硬件正常。

**第三轮排查**：添加 SYN6288 返回值读取，模块无任何回应。

**第四轮排查（根本性解决）**：仔细阅读 `\7.1、STM32F103C8T6应用程序\HARDWARE\SYN6288\syn6288.c` 官方参考代码，逐字节对比发现协议错误。

### 2.3 SYN6288 通信协议详解

#### 文本合成帧格式

```
字节位置    | 0     | 1   | 2        | 3   | 4            | 5..N | N+1
内容       | 0xFD  | 00  | Len      | 01  | 01|Music<<4 | 文本 | XOR
           | 帧头  | 长度高 | 长度低   | 编码 | 合成命令     | 数据 | 校验
```

#### 协议参数说明

| 字段 | 含义 | 正确值 |
|------|------|--------|
| 帧头 | 固定 `0xFD` | `0xFD` |
| 长度高 | 固定 0 | `0x00` |
| 长度低 | **文本字节数 + 3** | `HZ_Length + 3` |
| 编码 | `0x00`=ASCII, `0x01`=GB2312 | `0x01` |
| 合成命令 | `0x01` \| 背景音乐号 | `0x01` \| `Music << 4` |
| 文本 | GB2312 中文 + ASCII 数字 | |
| XOR | 从 Frame[0] 到最后一个文本字节异或 | |

### 2.4 四个致命协议错误

以播报"当前距离22厘米"为例，文本长度 = 8(中文) + 2(数字) + 4(中文) = 14 字节：

```
正确帧：FD 00 11 01 01 B5 B1 C7 B0 BE E0 C0 EB 32 32 C0 E5 C3 D7 DB
错误帧：FD 00 10 01 00 B5 B1 C7 B0 BE E0 C0 EB 32 32 C0 E5 C3 D7 26
          ^^    ^^    ^^                                            
        byte2   byte4                                              
```

| 错误 | 位置 | 错误值 | 正确值 | 后果 |
|------|------|--------|--------|------|
| 长度 | byte[2] | `14+2=16`(0x10) | **`14+3=17`(0x11)** | 模块拒绝接收（长度不符协议） |
| 命令 | byte[4] | `0x00` | **`0x01`** | 不是有效的"合成并播放"命令 |
| XOR范围 | 校验计算 | 跳过 Frame[0] | **包含 Frame[0]** | 校验错误，模块丢弃帧 |
| 停止命令 | 全帧 | 6字节文本格式 | **5字节芯片控制格式** | 停止命令也无效 |

### 2.5 芯片控制命令格式（与文本合成帧不同！）

```c
// 正确：5 字节芯片控制命令（对照参考项目 main.c）
// 格式：FD 00 02 [CMD] [XOR]
// XOR 范围：FD ^ 00 ^ 02 ^ CMD

static const uint8_t SYN_StopCom[]    = {0xFD, 0x00, 0x02, 0x02, 0xFD};
static const uint8_t SYN_SuspendCom[] = {0xFD, 0x00, 0x02, 0x03, 0xFC};
static const uint8_t SYN_RecoverCom[] = {0xFD, 0x00, 0x02, 0x04, 0xFB};
```

### 2.6 核心教训

**通信协议必须 100% 参照官方代码**。哪怕只差一个字节（长度 +3 vs +2、命令 0x01 vs 0x00），模块都会直接忽略整帧数据。SYN6288 是"静默失败"——不报错、不回应，让人误以为是硬件接线问题。

### 2.7 排查技巧

| 手段 | 用途 |
|------|------|
| USART 硬件回环测试 | 短接 TX/RX，确认 USART 确实在发数据 |
| 读取模块返回值 | SYN6288 收到有效命令会回发状态字节（如 `0x4A`=就绪） |
| 逐字节打印发送帧 | 用 printf 输出每一个字节，与官方代码对比 |
| BUSY 引脚检测 | 模块就绪时为高电平，始终为低说明模块异常 |

## 三、蜂鸣器调试复盘

### 3.1 问题现象

距离小于 20cm 时蜂鸣器正常滴滴响，大于 20cm 时一直响不停。

### 3.2 问题分析

两个根因，分两次修复：

#### 根因 1：SIG 模块电平逻辑反转

**硬件背景**：用户的蜂鸣器是 SIG 型四脚模块（3V3/GND/SIG/NC），控制逻辑是：
- `SIG = HIGH` → 蜂鸣器响
- `SIG = LOW` → 蜂鸣器停

**代码错误**：`buzzer.h` 中选了 `BUZZER_DRIVE_LOW` 模式：

```c
// BUZZER_DRIVE_LOW 模式（两脚直驱，错误）：
#define BUZZER_ON()   HAL_GPIO_WritePin(BUZZER_PORT, BUZZER_PIN, GPIO_PIN_RESET)  // LOW
#define BUZZER_OFF()  HAL_GPIO_WritePin(BUZZER_PORT, BUZZER_PIN, GPIO_PIN_SET)    // HIGH

// 结果：
// 报警关闭 → PB0=HIGH → SIG 模块收到 HIGH → 蜂鸣器响了（与预期相反）
```

**修复**：切换为 `BUZZER_DRIVE_HIGH` 模式：

```c
// BUZZER_DRIVE_HIGH 模式（SIG模块/三极管驱动，正确）：
#define BUZZER_ON()   HAL_GPIO_WritePin(BUZZER_PORT, BUZZER_PIN, GPIO_PIN_SET)    // HIGH→响
#define BUZZER_OFF()  HAL_GPIO_WritePin(BUZZER_PORT, BUZZER_PIN, GPIO_PIN_RESET)  // LOW→停
```

#### 根因 2：HC-SR04 超时返回旧值

**问题链路**：
1. 障碍物移走后，HC-SR04 收不到回波 → 超时
2. `HCSR04_GetDistance()` 在超时情况下返回的是**上一次测量的旧值** `g_distance`
3. 旧值可能仍在阈值内（比如上次测的 15cm）→ 程序以为障碍物还在 → 蜂鸣器一直响

**修复**：在读取前先检查是否有新数据：

```c
// 修复前：
distance = HCSR04_GetDistance();  // 超时时返回旧值

// 修复后：
if (!HCSR04_IsDone()) {
    distance = -1.0f;              // 超时：标记无效
} else {
    distance = HCSR04_GetDistance(); // 成功：取新值
}
```

### 3.3 两种蜂鸣器驱动的对应关系

| 模式 | 适用硬件 | `ON` 输出 | `OFF` 输出 |
|------|---------|----------|-----------|
| `BUZZER_DRIVE_HIGH` | SIG 模块 / 三极管驱动 | HIGH | LOW |
| `BUZZER_DRIVE_LOW` | 两脚直驱蜂鸣器（电流吸入） | LOW | HIGH |

```
│   ├── 0xBE 0xE0  "距" (GB2312: BEE0)
│   ├── 0xC0 0xEB  "离" (GB2312: C0EB)
│   ├── 0x32       "2"  (ASCII)
│   ├── 0x32       "2"  (ASCII)

│
├── SYN_FrameInfo(0, text)  Music=0(无背景音乐)
│   │

│   │

│   │   ecc = 0

│   │
│   ├── memcpy(Frame_Info+5, text, 14)  填入文本
│   ├── Frame_Info[19] = 0xDB           填入校验
│   │
│   └── HAL_UART_Transmit(&huart2, Frame_Info, 20)
│
├── SYN6288 模块收到数据
│   ├── 识别帧头 0xFD
│   ├── 读取长度 = 17
│   ├── 验证 XOR 校验 = 0xDB
│   ├── 解析编码 = GB2312
│   ├── 合成语音（内部 DSP 运算）
│   └── 驱动 SPK+/SPK- 输出模拟音频 → 喇叭发声
│
└── 主循环继续运行（非阻塞）
```

### 4.4 蜂鸣器报警工作流程

```
每 50ms 执行一次（tick-based 非阻塞定时）

┌─ 读取当前距离 distance ────────────────────┐
│                                              │
├─ distance < 0？（超时/无信号）                 │
│   └─ YES → BUZZER_OFF()  静音               │
│                                              │
├─ distance ≤ threshold？（在报警范围内）        │
│   └─ YES → BUZZER_TOGGLE()  PB0 电平翻转     │
│       ├─ 上次 LOW  → 这次 HIGH → 蜂鸣器响    │
│       ├─ 上次 HIGH → 这次 LOW  → 蜂鸣器停    │
│       └─ 50ms 周期 → 产生"滴滴滴"急促声      │
│                                              │
└─ distance > threshold？（超出报警范围）        │
    └─ YES → BUZZER_OFF()  PB0→LOW  静音      │
```

### 4.5 各模块引脚一览

| 模块 | STM32 引脚 | 配置 | 通信方式 |
|------|-----------|------|---------|
| OLED (SSD1306) | PB6(SCL) / PB7(SDA) | I2C1, 400kHz | I2C |
| HC-SR04 Trig | PA0 | GPIO 输出推挽 | 脉冲触发 |

## 五、通用调试方法总结

### 5.1 硬件通信调试三板斧

```
1. 回环自检 → 短接 TX/RX，发一段数据看能不能收回来 → 验证硬件层
2. 读取返回值 → 发命令后尝试从模块 RX 脚读回应 → 验证模块收到
3. 逐字节打印 → printf 输出发送的每一个字节，与参考代码对比 → 验证协议
```

### 5.2 常见错误模式

| 现象 | 可能原因 | 排查方法 |
|------|---------|---------|
| OLED 乱码 | 字库取模方式不匹配 | 逐字节对比参考项目字库 |
| OLED 全黑 | I2C 地址错了 / 线接反 | 查 SSD1306 地址是 0x3C 还是 0x3D |
| OLED 花屏 | 帧缓冲内存布局 bug | 打印发送的 GDDRAM 数据验证 |
| SYN6288 不响 | 协议帧格式错误 | 逐字节对比官方代码 |
| SYN6288 不响 | 波特率不匹配 | 确认模块默认波特率 |
| SYN6288 不响 | GND 没共地 | 万用表测 GND 连通性 |
| 蜂鸣器反相 | 驱动模式与硬件不匹配 | 查蜂鸣器是 SIG 型还是直驱型 |
| 蜂鸣器一直响 | 传感器超时返回旧值 | 加 `IsDone()` 超时判断 |
