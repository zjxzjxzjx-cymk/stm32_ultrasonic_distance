## 1. STM32 是什么

### 1.1 你手中的芯片

```
STM32F103C8T6
  │   │  │  ││
  │   │  │  │└─ 封装：T = LQFP (48脚)
  │   │  │  └── Flash大小：8 = 64KB
  │   │  └───── 系列：103 = 性能型
  │   └──────── 核心：F1 = Cortex-M3 (72MHz)
  └──────────── ST 意法半导体
```

这是一颗 **72MHz 主频、64KB Flash、20KB SRAM** 的 MCU。

### 1.2 MCU vs CPU vs SoC

| | CPU | MCU | SoC |
|--|-----|-----|-----|
| 全称 | Central Processing Unit | MicroController Unit | System on Chip |
| 比喻 | 人的大脑 | 大脑+手脚+感官的完整机器人 | 一台完整的微型电脑 |
| 包含 | 仅运算核心 | CPU + RAM + ROM + IO + 定时器 + 外设 | CPU + GPU + RAM + 所有外设 |
| 例子 | Intel i7 | STM32F103、Arduino | 树莓派的芯片 |

你手中的 STM32 是 **MCU**，它把 CPU、内存、GPIO、定时器、串口等全部集成在一颗芯片里。

### 1.3 HAL 库是什么

```
你想让 PB0 输出高电平，可以直接写：
  GPIOB->BSRR = 0x00000001;        // 直接操作寄存器

但 ST 公司提供了 HAL 库：
  HAL_GPIO_WritePin(GPIOB, GPIO_PIN_0, GPIO_PIN_SET);  // 函数调用

HAL = Hardware Abstraction Layer（硬件抽象层）
  作用：把繁琐的寄存器操作包装成易读的函数
  代价：执行速度稍慢一点（微秒级差异，本项目不影响）
```

本项目全部使用 HAL 库。

## 2. GPIO 通用输入输出

### 2.1 什么是 GPIO

每个 GPIO 引脚就是一个能**被程序控制**的电路连接点。你可以让它：

| 方向 | 做什么 | 本项目哪里用到 |
|------|--------|---------------|
| **输出** | 输出高(3.3V)或低(0V)电平 | PB0 控制蜂鸣器、PA0 给 HC-SR04 发 TRIG |
| **输入** | 读取外部引脚的电平 | PB12/PB13 读按键状态 |

### 2.2 输出模式：推挽 vs 开漏

```
推挽输出 (Push-Pull) — 本项目用这个
   3.3V ──→ [P-MOS] ──┐
                       ├──→ 引脚
   GND  ──→ [N-MOS] ──┘
   
   输出高：P-MOS 导通，输出 3.3V（能提供电流）
   输出低：N-MOS 导通，输出 0V（能吸收电流）

开漏输出 (Open-Drain)
   只有 N-MOS，没有 P-MOS。
   输出低：N-MOS 导通到 GND
   输出高：N-MOS 截止，引脚"悬空"（需要外部上拉电阻才能到高电平）
   用途：I2C 总线、多设备共享的信号线
```

### 2.3 内部上下拉电阻

```
CubeMX 配了 PB12 = Input + Pull-up：

芯片内部：
   3.3V ──[40KΩ电阻]──┬──→ 引脚 PB12 ──→ 按键 ──→ GND
                      │
                   读到电平

   按键没按：引脚被 40KΩ 拉到 3.3V → 读到 1
   按键按下：引脚直连 GND，电流 = 3.3V/40KΩ = 82μA → 读到 0
```

没有这个上拉电阻，按键没按时引脚"悬空"，电平不确定——一会儿读0一会儿读1。

### 2.4 BSRR 寄存器（STM32 的绝技）

```
GPIOB->BSRR 是一个 32 位寄存器：

   位31 ────────────── 位16 │ 位15 ────────────── 位0
   ┌────────────────────────┬────────────────────────┐
   │ Bit Reset Register     │ Bit Set Register       │
   │ (写1=对应引脚变低电平) │ (写1=对应引脚变高电平) │
   └────────────────────────┴────────────────────────┘

例：
  GPIOB->BSRR = 0x00000001;     // bit0写1 → PB0变高
  GPIOB->BSRR = 0x00010000;     // bit16写1 → PB0变低
  GPIOB->BSRR = 0x00010001;     // 同时：PB0变低！(Reset优先级高于Set)
```

**为什么用 BSRR 而不用 ODR（输出数据寄存器）？**

ODR 需要"读-改-写"三步操作：如果你同时修改 PB0 和 PB3，而中断恰好在读和写之间改动了 PB3，那你的写操作会覆盖掉中断的更改。BSRR 是原子操作，不会出现这个"竞态条件"。

## 3. 定时器

### 3.1 定时器 = 一个自动计数的电路

```
定时器本质上是一个"计数器"：

  时钟源 ──→ [预分频器] ──→ [计数器 CNT] ──→ 计数值
                ÷N             0→1→2→...

计数值到达某个数后可以：
  - 归零(ARR) → 产生溢出中断
  - 等于比较值(CCR) → 改变引脚电平(输出比较)
  - 引脚跳变 → 锁存当前计数值(输入捕获)
```

### 3.2 本项目的 TIM2 配置

```
时钟树：
  HSE 8MHz → PLL ×9 = 72MHz → SYSCLK → AHB → APB1 定时器时钟 = 72MHz

TIM2 配置：
  Prescaler = 72-1 = 71    → 计数频率 = 72MHz/72 = 1MHz
  ARR = 65535              → 最大计数值
  
结果：每 1μs CNT 值 +1，最大连续计数 65.535ms
```

### 3.3 为什么 ARR 用 65535？

TIM2 是 16 位定时器，最大值就是 2^16 - 1 = 65535。配合 1MHz 计数频率，能测的最大时间是 65.535ms，对应 HC-SR04 最大距离 ≈ 1130cm（远超模块的 400cm 极限）。

### 3.4 什么是"自动重装载"

计数器到 65535 后：
- **禁止重装载**：计数器停在 65535 不动（本项目用的）
- **使能重装载**：计数器自动归零，重新开始（产生周期信号时用）

## 4. 输入捕获

### 4.1 硬件自动帮你记时间

```
场景：PA1(ECHO) 引脚突然变高 → 变低，测量高电平持续多久。

笨办法：
  死循环读电平 → 读到1开始计时 → 读到0停止。
  问题：CPU 不能做别的事。

聪明办法（输入捕获）：
  配置 TIM2 CH2 → PA1 边沿跳变时
  → 硬件自动把当前 CNT 值存入 CCR2 寄存器
  → 发中断通知 CPU
  → CPU 只在中断里处理，其他时间做别的事
```

### 4.2 上升沿 → 下降沿 的切换

```
时间线：
  ECHO ──┐        ┌──────────┐
         └────────┘          └────────

  TIM2 中断触发：
    ① T1=1234 (上升沿) → 记录时间 → 切换为下降沿捕获
    ② T2=15000 (下降沿) → 脉宽 = 15000-1234 = 13766μs
                        → 距离 = 13766 / 58 ≈ 237cm
                        → 切换回上升沿，准备下一轮
```

切换边沿用 `__HAL_TIM_SET_CAPTUREPOLARITY()` 宏，本质是改写 TIM2 的 CCER 寄存器中的 CPOL 位。

### 4.3 距离公式推导

```
声速 340m/s = 34000cm/s = 0.034cm/μs

ECHO 高电平时间 t (μs) = 超声波从发射到接收的往返时间

单向距离 = t × 0.034 / 2 = t × 0.017 = t / 58.82

简化：距离(cm) ≈ t / 58
```

## 5. 中断

### 5.1 中断是什么

```
中断 = "CPU 正在做 A，突然 B 事件发生，CPU 暂停 A 去处理 B，处理完回来继续 A"

类比：
  你在看书（A），手机响了（中断B）
  → 你放下书去接电话（ISR）
  → 挂电话后继续看书（返回 A）
```

**如果没有中断，CPU 必须反复"轮询"B 是否发生——浪费算力。**

### 5.2 中断的三个关键参与者

```
事件源      →  中断控制器    →  CPU
(TIM2溢出)    (NVIC)          (Cortex-M3)

1. 事件源：TIM2 计数器溢出产生"中断请求"
2. NVIC（嵌套向量中断控制器）：决定"先处理谁"
3. CPU：跳转到对应的中断服务函数
```

### 5.3 中断优先级

```
STM32F1 优先级分组 NVIC_PRIORITYGROUP_4：
  4 位全部用于抢占优先级（0~15，小数字=高优先）

  抢占优先级 0 → 可以打断抢占优先级 1 的中断
  抢占优先级 1 → 不能打断抢占优先级 0 的中断

本项目：
  TIM2: 抢占优先级 0（最高，测距不能等）
  SysTick: 抢占优先级 15（最低，系统滴答不着急）
```

### 5.4 HAL 中断机制

```
硬件中断 → 启动文件(.s)中的向量表 → TIM2_IRQHandler()

TIM2_IRQHandler() {           // 你在 stm32f1xx_it.c 写的
    HAL_TIM_IRQHandler(&htim2);  // HAL 帮你读状态寄存器、清标志
}

HAL_TIM_IRQHandler() 内部根据中断来源调用：
  → HAL_TIM_IC_CaptureCallback()    对 CC 事件
  → HAL_TIM_PeriodElapsedCallback()  对溢出事件
```

### 5.5 中断中需要 `volatile` 关键字

```c
// hcsr04.c
static volatile float g_distance = -1.0f;

/*
不加 volatile：
  编译器看到 main.c 中读取 g_distance，会优化"放到寄存器里"。
  但 g_distance 在中断中被修改——编译器不知道！
  导致 main.c 永远读到寄存器中的旧值。

加了 volatile：
  告诉编译器"每次都从内存重新加载，别优化"。
*/
```

## 6. I2C 通信

### 6.1 I2C 是什么

```
I2C (Inter-Integrated Circuit) = 两线串行总线
  
  只需要 2 根线：
    SCL — 时钟线（主机发）
    SDA — 数据线（双向）

  一根总线上可以挂多个设备，每个设备有唯一地址。
```

### 6.2 I2C 通信时序

```
一帧数据的格式：

  [START] [地址+R/W] [ACK] [数据0] [ACK] ... [数据N] [ACK/NACK] [STOP]

START：SCL高电平时，SDA从高变低     ─── SCL ───╲
                                   ─── SDA ───╲

数据：SCL低时SDA可变化，SCL高时SDA必须稳定
       SCL  ___/‾‾‾\___/‾‾‾\___
       SDA  ___X_____X_____X___  ← SCL高时采样

ACK：第9个SCL脉冲时，接收方拉低SDA
     主机发完8位数据后释放SDA → 从机拉低SDA → 主机检测到低 = ACK

STOP：SCL高电平时，SDA从低变高     ─── SCL ───╱
                                   ─── SDA ───╱
```

### 6.3 SSD1306 的 I2C 地址

```
7位地址： 0 1 1 1 1 0   = 0x3C

HAL 库参数需要左移1位：0x3C << 1 = 0x78
  （HAL_I2C_Master_Transmit 内部会在最低位加 R/W）

本项目 I2C 速率设置为 400kHz (Fast Mode) 以提高 OLED 刷新速度。
```

## 7. SSD1306 OLED 驱动

### 7.1 屏幕组织方式

```
128 × 64 像素的 OLED 被分成 8 个"页"(Page)：

  Col:  0 ───── 127
  Page0 █████████████  ← 8 像素高（第0-7行）
  Page1 █████████████  ← 8 像素高（第8-15行）
  ...
  Page7 █████████████  ← 8 像素高（第56-63行）

每页 = 128列 × 1字节
1字节 = 8像素垂直排列，bit0=最上像素（共阴-列行式）
```

### 7.2 页寻址模式（直接写 GDDRAM）

```
本项目采用的方案（对照野火 bsp_i2c_oled.c）：

方案：直接写 GDDRAM（Graphics Display RAM）
  - 所有绘制操作立即通过 I2C 发送到 OLED
  - 无需在 MCU 内存中维护帧缓冲（省内存）

优点：
  - 代码简单，没有帧缓冲和"刷新"的分离
  - 不会出现内存布局 bug（帧缓冲大小/偏移错误）
  - 省 1KB SRAM（128×8字节）

缺点：
  - I2C 通信次数更多（每次像素修改都走 I2C）
  - 对于本项目"一次写整屏幕"的使用场景，差异不大

核心函数对照参考项目：
  写命令: HAL_I2C_Master_Transmit(0x3C<<1, buf, 2, timeout)
         前字节=0x00(命令), 后字节=命令值
  写数据: HAL_I2C_Master_Transmit(0x3C<<1, buf, len, timeout)
         前字节=0x40(数据), 后接像素数据

注意：HAL 的 I2C 地址参数需要 7位地址左移1位
      0x3C << 1 = 0x78
```

### 7.3 OLED 初始化序列

SSD1306 上电后需要一串"咒语"才能工作：

```
0xAE  关闭显示（配置期间先关）
0xD5+0x80  时钟设置
0xA8+0x3F  复用比 64（屏幕高度）
0x8D+0x14  开电荷泵（3.3V下必须！否则屏幕全黑）
0xA1+0xC8  镜像方向设置
0x81+0xCF  对比度
0xAF  开显示
```

### 7.4 字库原理

```
字符 '0' 的 6×8 像素表示：

    列0  列1  列2  列3  列4  列5
行0  ·   ██   ██   ██   ██   ·
行1 ██   ·    ·    ·    ·   ██
行2 ██   ·    ·    ·    ·   ██
行3 ██   ·    ·    ·    ·   ██
行4 ██   ·    ·    ·    ·   ██
行5 ██   ·    ·    ·    ·   ██
行6 ██   ·    ·    ·    ·   ██
行7  ·   ██   ██   ██   ██   ·

编码为6字节（每列1字节，bit0=行0, bit7=行7）：
  { 0x3E, 0x51, 0x49, 0x45, 0x3E }

 0x3E = 0b00111110 → 行0=0,行1=1,行2=1,...,行6=1,行7=0
```

## 8. UART 通信

### 8.1 UART 和 I2C 的区别

| | UART | I2C |
|--|------|-----|
| 线数 | 2 根 (TX+RX) | 2 根 (SCL+SDA) |
| 时钟 | 无（异步，双方约定波特率） | 有（主机提供 SCL） |
| 拓扑 | 点对点 | 多设备总线 |
| 速度 | 灵活（9600~数Mbps） | 标准100k/400k |
| 本项目用 | SYN6288 语音 | OLED 显示 |

### 8.2 异步通信原理

```
因为没有时钟线，接收方怎么知道"什么时候采样"？

约定波特率 = 9600：
  → 每个 bit 持续 1/9600 ≈ 104μs
  
发送方按 9600 的节奏发，接收方按 9600 的节奏收。
双方各用自己的"时钟"计时，不需要共享时钟线。
```

### 8.3 数据帧格式

```
常见的 "9600, 8N1"：

  [起始位] [8个数据位] [无校验] [1个停止位]
     0      D0-D7                 1

空闲：线路上保持高电平
起始位：拉低 1 bit 时间（告诉接收方"数据来了"）
数据位：8 bit，LSB 先发
停止位：拉高 1 bit 时间（告诉接收方"数据结束"）

所以：每个字节实际上要发 10 个 bit (1+8+0+1)
    有效速率 = 9600 × 8/10 = 768 字节/秒
```

## 9. 通信协议帧

### 9.1 为什么需要"帧"

```
UART 传输的是原始字节流。
接收方收到的就是一串字节：FD 00 11 01 00 B5 B1 C7 B0 BE E0 C0 EB ...

问题：
  1. 收到 FD，它是"帧头"还是"巧合的数据"？
  2. 这条消息有多长？何时结束？
  3. 传输过程中出错了怎么知道？

答案：协议帧。
  帧头 = 标记消息起点
  长度 = 告诉接收方消息有多长
  校验 = 检测传输错误
```

### 9.2 SYN6288 协议帧（对照官方参考项目）

```
文本合成帧（SYN_FrameInfo）：

[0xFD] [0x00] [Len] [0x01] [0x01] [GB2312文本...] [XOR]
 ─┬─    ─┬─    ─┬─   ─┬─   ─┬─    ────┬───────    ─┬─
 帧头   长度高 长度  编码  合成命令  GB2312负载    校验
        (恒为0)      (GB2312)  (播放)

Len = 文本字节数 + 3
合成命令 = 0x01 | (Music << 4), Music=0为无背景音乐

XOR 校验范围：从 Frame[0] 到最后一个文本字节（包含 0xFD 帧头！）
  参考项目代码：for(i=0; i<5; i++) ecc = ecc ^ Frame_Info[i];
  这说明 0xFD、0x00、Len、0x01、Cmd 这5个字节都参与 XOR

例：播报"当前距离30厘米"
  "当前距离" = B5 B1 C7 B0 BE E0 C0 EB (8字节 GB2312)
  "30"       = 33 30                   (2字节 ASCII)
  "厘米"     = C0 E5 C3 D7             (4字节 GB2312)
  负载 = 14 字节，Len = 14+3 = 17 = 0x11

  帧 = FD 00 11 01 01 B5B1C7B0BEE0C0EB 3330 C0E5C3D7 [XOR=0xDB]

---

芯片控制命令（停止/暂停/恢复/状态查询）：

格式：FD 00 02 CMD XOR   (5字节，注意与文本合成帧不同！)
XOR = FD ^ 00 ^ 02 ^ CMD

例如停止合成：
  SYN_StopCom = {0xFD, 0x00, 0x02, 0x02, 0xFD}
  XOR = FD^00=FD, FD^02=FF, FF^02=FD
```

### 9.3 XOR 校验讲解

```
XOR = 异或运算（Exclusive OR）

真值表：      口诀：
  0 ^ 0 = 0   相同得 0
  0 ^ 1 = 1   不同得 1
  1 ^ 0 = 1
  1 ^ 1 = 0

核心性质：
  A ^ A = 0         任何数异或自己 = 0
  A ^ 0 = A         任何数异或 0 = 自己
  A ^ B ^ A = B     异或满足交换律

计算过程（逐字节异或，从 Frame[0]=FD 开始）：
  Data:  0xFD  0x00  0x11  0x01  0x01  0xB5  0xB1  ...
  XOR累加：0xFD
           FD ^ 00 = FD
           FD ^ 11 = EC
           EC ^ 01 = ED
           ED ^ 01 = EC
           EC ^ B5 = 59
           59 ^ B1 = E8
           ...继续到最后一个文本字节...
  最终结果 = 0xDB

接收方把所有字节 XOR 一遍，如果结果 = 0，说明传输没有单bit错误。
```

### 9.4 GB2312 编码

```
GB2312 是中国制定的简体中文编码标准。

每个汉字 = 2 字节，每字节范围 0xA1~0xF7

例：
  "当" = 0xB5 0xB1
  "前" = 0xC7 0xB0
  "距" = 0xBE 0xE0
  "离" = 0xC0 0xEB

C 语言中用十六进制数组存储：
  uint8_t text[] = {0xB5,0xB1, 0xC7,0xB0, 0xBE,0xE0, 0xC0,0xEB};
  // 每个汉字2字节，直接用 GB2312 编码值
  // 注意：这种方式与源文件的 UTF-8/GB2312 编码无关——
  // 编译器看到的是 0xB5 这个数字，不是中文"当"字本身
```

## 10. 按键消抖

### 10.1 物理世界的"弹跳"现象

```
理想情况：
  按下 → 电平立刻从 1 变 0
  松开 → 电平立刻从 0 变 1

实际情况（用示波器看）：
  按下 → 3.3V ▁▂▃▄▅▆▇█▇▆▅▄▃▂▁ 0V   (5~20ms弹跳)
  松开 → 0V   ▁▂▃▄▅▆▇█▇▆▅▄▃▂▁ 3.3V (5~20ms弹跳)

如果不消抖，按一下键可能被读成几十次"按下-松开"。
```

### 10.2 延时消抖法

```
算法：
  1. 检测到低电平（疑似按下）
  2. 等 20ms
  3. 再读一次 → 还是低 = 真按下
                → 已经是高 = 抖动，忽略

为什么是 20ms？
  - 机械按键弹跳时间通常在 5~20ms
  - 20ms 能覆盖绝大多数情况
  - 太短(5ms)可能消不干净
  - 太长(50ms)按键反馈迟钝
```

### 10.3 更高级的消抖方法

本项目用的是最简单的"延时消抖"。更高级的：
- **多次采样法**：每 1ms 读一次，连续 10 次低电平 = 真按下（不用阻塞 20ms）
- **硬件消抖**：在按键两端并联 0.1μF 电容 + 施密特触发器（硬件解决，程序最简洁）

## 12. 阻塞 vs 非阻塞

### 12.1 阻塞写法的问题

```c
// 阻塞写法：HAL_Delay 让 CPU "傻等"
Buzzer_BeepShort(100);
// 这 100ms 内 CPU 躺在 HAL_Delay 里不停数数，
// 不能做任何其他事。如果此时有按键按下？不理你。


// 更糟的阻塞写法：
while (按键没松手) { }   // 一直卡着，CPU 100% 占用但啥也没干
```

### 12.2 非阻塞写法（本项目报警部分）

```c
// 非阻塞写法：记录时间，到了就做动作，没到就往下走
if (distance <= threshold) {
    if (now - alarm_tick >= 100) {  // 距离上次翻转超过 100ms
        alarm_tick = now;           // 更新时间
        BUZZER_TOGGLE();            // 翻转蜂鸣器
    }
}
// 不阻塞！不管翻转没翻转，程序都继续往下走。
```

### 12.3 设计要点

```
非阻塞设计的核心思路：
  1. 用"状态"记录当前情况（BUZZER_ON/OFF 就是一个状态）
  2. 用"时间戳"记录上次动作时间（alarm_tick）
  3. 每次循环检查"距离上次动作过了多久"
  4. 时间到了就执行，没到就跳过

好处：
  - 一个循环里可以同时监控多个任务
  - 哪个任务条件满足就执行哪个
  - 谁都不耽误谁
```

## 13. 主程序结构

### 13.1 本项目用的是"前后台系统"

```
┌──────── 后台（中断） ──────────┐
│ TIM2 中断 → 输入捕获 → 算距离   │   后台自动运行，无需主循环操心
│ SysTick → HAL_Delay 计时        │
└─────────────────────────────────┘

┌──────── 前台（主循环） ────────┐
│ while(1) {                      │
│   1. 按键扫描                   │   前台每 200ms 跑一轮
│   2. 触发测距                   │
│   3. OLED 刷新                  │
│   4. 报警判断（非阻塞）          │
│   5. 语音播报（非阻塞）          │
│   HAL_Delay(200);  ← 200ms 间隔 │
│ }                               │
└─────────────────────────────────┘
```

### 13.2 主循环 5 步详解

```
while (1) {
    ① 按键扫描 — 阻塞但可接受（等了也就几百ms，按键后本来也没事做）
    
    ② 触发测距 — 发 TRIG 脉冲 → 等 ECHO 中断完成（最多等 100ms）
    
    ③ OLED 刷新 — 清缓冲 → 画内容 → 一次性送屏幕（纯计算 + I2C 发送）
    
    ④ 报警判断 — 非阻塞！用时间戳驱动，不影响其他功能
    
    ⑤ 语音播报 — 非阻塞！条件满足才播，不影响主循环节奏
    
    HAL_Delay(200);  // 控制整体刷新速率 = 5Hz
}
```

### 13.3 模块化设计

```
项目文件结构：

main.c     ← 只管"什么时候做什么"，不关心"怎么做"
  │
  ├── buzzer.c/h   ← 只管蜂鸣器响不响
  ├── key.c/h      ← 只管按键检测和阈值管理
  ├── hcsr04.c/h   ← 只管超声波测距
  ├── ssd1306.c/h  ← 只管 OLED 显示
  └── syn6288.c/h  ← 只管语音播报

每个模块：
  - .h 文件 = "接口"（告诉别人你能做什么）
  - .c 文件 = "实现"（你具体怎么做，外部不需要知道）

好处：
  改蜂鸣器 → 只改 buzzer.c，其他 5 个模块不受影响
  换 OLED  → 只改 ssd1306.c，main.c 不用动
```

## 14. 硬件通信调试方法论（实战总结）

### 14.1 调不通时的三步排查法

```
第1步：硬件层 → 信号有没有真的发出去？
  方法：USART 回环测试（短接 TX/RX）
  结果：PASS → 硬件正常 → 继续第2步
        FAIL → 引脚配置/接线有问题

第2步：模块层 → 模块有没有收到并回应？
  方法：发命令后尝试读取模块返回值
  结果：有回应（如 0x4A）→ 模块收到 → 继续第3步
        无回应 → 接线/波特率/协议有误

第3步：协议层 → 数据帧格式是否正确？
  方法：printf 逐字节打印发送帧 → 与官方参考代码对比
  结果：每一字节都符合 → 协议正确
        有差异 → 找到差异点，修复
```

### 14.2 SYN6288 通信协议常见错误（实战教训）

本项目中 SYN6288 曾因为 4 个协议错误导致完全不响应：

| 错误 | 错误值 | 正确值 | 怎么发现的 |
|------|--------|--------|-----------|
| 长度偏移 | `text+2` | **`text+3`** | 对照参考项目 `SYN_FrameInfo()` |
| 合成命令 | `0x00` | **`0x01`** | 对照 `Frame_Info[4] = 0x01` |
| XOR 范围 | 跳过 Frame[0] | **包含 Frame[0]** | 对照 `for(i=0;i<5;i++)` |
| 停止命令 | 6字节文本格式 | **5字节芯片控制格式** | 对照 `SYN_StopCom[]` |

**关键教训**：通信协议差一个字节，模块直接忽略整个帧，无任何报错（"静默失败"）。

### 14.3 OLED 显示乱码常见原因

| 现象 | 可能原因 | 排查方法 |
|------|---------|---------|
| 字符形状不对但可辨认 | 字库取模方式不匹配 | 逐字节对比参考项目字库 |
| 全屏花斑/乱点 | 帧缓冲内存布局 bug | 打印发送的 GDDRAM 数据验证 |
| 屏幕全黑 | I2C 地址错误 / 初始化序列不全 | 检查地址 0x3C vs 0x3D / 对比 init 序列 |
| 部分行不显示 | 页寻址模式下页地址设错 | 检查 SetPos 的 0xB0+page 是否正确 |

### 14.4 蜂鸣器不响/反转

- **SIG 型模块**：SIG=HIGH 响，代码选 `BUZZER_DRIVE_HIGH`
- **直驱两脚蜂鸣器**：PB0=LOW 吸入电流响，代码选 `BUZZER_DRIVE_LOW`
- 选错模式会导致"该响时不响，该停时一直响"

### 14.5 调试工具速查

| 工具 | 用法 | 场景 |
|------|------|------|
| USB-TTL + 串口助手 | 看 printf 输出 | 验证程序逻辑、打印帧数据 |
| 万用表蜂鸣档 | 测通断 | 检查杜邦线是否接触良好 |
| USART 回环 | 短接 TX/RX 自收发 | 验证 USART 硬件是否正常 |
| BUSY 引脚 | 读电平 | 判断模块是否就绪 |
| 模块 LED | 目视 | 判断模块是否上电 |
